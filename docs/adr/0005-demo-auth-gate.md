# ADR-0005 — Demo-auth gate (login page + session cookie)

Status: accepted (2026-07-12)

## Context / forces

Filature has shipped with no authentication (AR-001): every route is open,
mitigated only by binding to a trusted interface behind a reverse proxy. The
user now wants to deploy **public demo instances** that must not be wide open,
without turning the single-operator tool into a multi-user auth system.

Forces:
- **Minimal, self-hosted.** One operator, one credential. No user store, no
  roles, no registration, no PII. A full IdP/session-store is overkill.
- **Must not regress the test suites.** The e2e/it suites drive `web::router`
  directly; auth must not force every existing test through a login flow.
- **Deployed via Docker with no `filature.toml` in the image** — the credential
  has to be settable purely from env.
- **The app terminates no TLS** (an external proxy does) and can't reliably know
  the outer scheme.

## Decision

A lightweight **demo-auth gate** in the driving adapter (`web::auth`):

1. **Separate `AuthConfig`, not a field of `Config`.** Loaded by its own figment
   extract on the `[auth]` table (same TOML+env sources), so the feature is
   purely additive — zero churn to the existing `Config` literals in tests.
2. **Enforcement is an outer layer** applied in `main.rs` around
   `web::router(...)`, never inside it. Production always wraps it; the
   `web::router`-based test suites don't, so they stay open and unchanged.
3. **Always active in production.** A missing `[auth]` table fails the boot,
   rather than silently serving an open instance.
4. **Default-deny middleware, allowlist = `/login` only.** Every other path —
   `/static` included — redirects to `/login` without a valid session cookie, so
   nothing of the app is served unauthenticated except the self-contained login
   page.
5. **Vetted crypto, no hand-rolled auth.** argon2id (`argon2` crate) for the
   password (`password_hash` in config, generated by `filature hash-password`);
   the session token is 32 bytes from `OsRng`.
6. **Per-boot shared session token, held in memory.** One value for the single
   operator; a restart invalidates sessions. Keeps zero secret material in
   config and needs no session store. Cookie is `HttpOnly; SameSite=Lax`.

## Rejected alternatives

- **HTTP Basic Auth.** Simplest (no page/template), but no clean logout, an ugly
  native popup, and credentials sent on every request. The user asked for a
  login page.
- **Auth inside `web::router` (per-request `AppState.auth`).** Would force all
  existing e2e/it tests through the gate (mass 302s) or need auth wiring in every
  test's `AppState`. The outer-layer choice avoids both.
- **`auth` as a required field on `Config`.** Breaks the 4 `test_config`
  literals and couples the feature to the core config struct.
- **Persistent/per-session tokens + session store.** Overkill for one demo user;
  the per-boot shared token is enough and simpler.
- **`Secure` cookie flag always on.** Would break plain-HTTP local runs, and the
  app can't detect TLS behind the proxy. Omitted deliberately (AR-006).

## Consequences

- Closes AR-001 for demo deployments (authentication now enforced), but it is
  **authentication only** — no per-user authz/IDOR model, by design.
- Known residual risks recorded as **AR-005** (no brute-force/rate limiting,
  mitigated by argon2's cost) and **AR-006** (no `Secure` flag; single shared
  per-boot token).
- Promote to full authz + per-session tokens if the app ever becomes multi-user
  or is exposed beyond a controlled demo.
