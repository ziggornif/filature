| TD-009 | **Create/edit form-validation errors (422 blank name, 409 duplicate name) were not surfaced in the UI** for materials/locations create+edit ‚Äî handlers returned unwrapped text and htmx never swaps non-2xx bodies, so mistakes failed silently. | Single-user, self-hosted; low real-world friction. Deferred until the `response-targets` pattern was needed for more than the one locations-delete control. | **Resolved 2026-07-11** ‚Äî added `web::router::form_error()`: renders the message through Tera (`_form_error.html`, HTML-escaped) and tags the response `HX-Reswap: innerHTML`. Every create/edit control (materials + locations add form and inline edit inputs) opts into `hx-ext="response-targets"` + `hx-target-error="#‚Ä¶-msg"`, routing 422/409/404 to a per-page `#materials-msg`/`#locations-msg` slot. `HX-Reswap: innerHTML` normalizes the swap so both the add form's `beforeend` and the edit inputs' `outerHTML` fill the slot cleanly (no nesting/wipe). `render_row` appends an OOB clear so a successful add/edit dismisses a stale error. New i18n keys (`materials.error.{invalid,duplicate,not_found}`, `locations.error.{invalid,not_found}`, en+fr). `.form-error:empty{display:none}` collapses the empty slot. Verified end-to-end in a real browser (CDP on the live page): materials 409-duplicate, materials 422-blank-edit (row intact), OOB clear-on-success, locations 422-blank-edit (row intact); plus e2e HTTP-contract tests (status + `HX-Reswap` + escaped localized body + name interpolation) and render-level wiring guards. Note: the add form's blank-name 422 is unreachable via the UI (`required` blocks it); the reachable add-form case is the 409 duplicate. |
| TD-003 | **`update()` of an unknown material id was a silent 200 no-op** (materials slice). Locked by a stub unit test, an `it_materials` Postgres test, and an `e2e_materials` `PUT /materials/{unknown}` ‚Üí 404 test. |

| TD-011 | **`Money` is not quantized to cents ‚áí Stock Value can differ between the dashboard KPI and the spools-list stat, and both show long decimal tails.** The spools `stock_value` sums `(remaining::numeric / net::numeric) * price` in Postgres NUMERIC; the dashboard computes the same sum as a domain fold (`f64` `Grams::ratio_of` ‚Üí `Decimal::try_from` ‚Üí √ó price, in `DashboardOverview::from_rows`). For non-terminating ratios (net not a divisor of remaining, e.g. net=750) the two produce `Decimal`s that differ in trailing digits, and `Money` compares/displays by exact `Decimal` with no cent rounding, so the two "Valeur du stock" figures can disagree and both render an ugly tail (e.g. `33.333333‚Ä¶`). Flagged by the slice-05 whole-branch review as conflicting with the "must agree" acceptance criterion. | Personal-scale, single-user; at the common net=1000 g the two paths agree exactly and the divergence is otherwise sub-cent/far-decimal. The ugly-tail display is pre-existing on the spools list, not introduced by the dashboard. User chose (2026-07-10) to log rather than block the slice merge. | The two Stock Value numbers can visibly differ on any spool whose net isn't a clean divisor; both look unpolished (unrounded). A later automated equality assertion at a fractional ratio would fail. | **Resolved 2026-07-10** ‚Äî `Money`'s `Display` now renders to 2 dp, rounded (`write!(f, "{:.2}", self.0.round_dp(2))` ‚Äî `{:.2}` alone truncates). Display-only: stored `Decimal` keeps full precision. Fixes the raw-tail render (`1.760000‚Ä¶` ‚Üí `1.76`) and makes the spools NUMERIC stat and the dashboard f64-fold KPI read identically, since both collapse to the same cents. Locked by `money_displays_to_two_decimals_rounded` (high-scale‚Üícents, whole‚ÜíX.00, round up/down, non-destructive). Applies everywhere `Money` is shown (stock value both screens, prices). |
