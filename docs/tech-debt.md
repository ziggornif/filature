## Active debt

| ID | Description | Why accepted | Cost if left | Priority |

| TD-001 | **Home-grown i18n** (`web/i18n.rs`): ~40-line JSON-catalog lookup with locale→default→key fallback. No pluralization, gender, or `{var}` interpolation. | Does exactly what ADR-0001 needs today (key→string); zero deps; YAGNI. Reviewed crates: **fluent-templates** (Mozilla Fluent/ICU — plurals, gender, interpolation, `.ftl` files; heaviest, most capable) and **rust-i18n** (compile-time macro, JSON/YAML/TOML catalogs, `t!()` macro; lighter but macro-bound). | When a real string needs a plural or an interpolated value, the custom layer must grow or be swapped — a swap touches every template's `t()` call. | low — revisit when the first pluralized/interpolated string appears (likely a slice with counts, e.g. "N bobines"). |
| TD-002 | **Basic observability**: startup uses `tracing` with a simple subscriber; no structured request spans, no metrics, no log levels wired to config. | Foundation needs *a* real logger (not `println!`); full observability is not needed pre-v1. | Debugging a deployed instance is coarse until request-scoped tracing + levels land. | med — add `tower-http` trace layer spans + config-driven level when the app is actually deployed to k3s. |
| TD-003 | **`update()` of an unknown material id is a silent 200 no-op** (materials slice). The SQLx `MaterialRepository::update` doesn't check `rows_affected`, so `PUT /materials/{unknown-id}` runs a 0-row UPDATE and returns a 200 row fragment as if it persisted. The in-memory stub instead returns `Backend("no such id")` (→500), so stub-backed tests don't catch the divergence. | Not reachable from the happy-path htmx UI (ids always come from a rendered row); no data loss; final review cleared it as deferrable. | A future non-UI caller (API, bulk edit) gets a false success; stub/prod behaviour disagree. | med — add a `RepositoryError::NotFound` variant, check `rows_affected == 0` in the SQLx `update`, map to **404** in the web `edit` handler, and align the stub. |
| TD-005 | **Raw internal error strings returned to the client** (materials **and spools** web handlers). 500 arms return `sqlx`/`tera` error text in the response body (info disclosure). The spools slice (03a) mirrors the same pattern across its handlers. | v1 is single-user, self-hosted, no auth — the only client is the trusted operator; practical exposure is nil. | Leaks backend/SQL/template internals once the surface is ever exposed beyond the operator. | low — map 500s to a generic message, log the detail server-side; do this if/when the app is exposed beyond localhost. |
| TD-006 | **`spools` list adapter maps 9 positional args with no swap-guard test** (`crates/app/src/persistence/spools.rs` `to_list_item`; `crates/app/tests/it_spools.rs`). `net_weight`/`remaining_weight` (and other same-typed fields) are adjacent positional args; a future transposition would compile and, since tests use spools where `remaining == net`, would not be caught. `it_materials.rs` has a deliberate distinct-value swap-guard that wasn't carried over. Correct today. | Slice shipped green; the bug is latent, not present. | A later column/arg transposition silently corrupts displayed weight/ratio/length with no failing test. | med — add a list test asserting distinct net vs remaining after a draw-down; consider a struct-literal mapping to remove the positional hazard. |
| TD-007 | **`spools` slice minor polish** (03a). (a) `update` handler's `UnknownMaterial`→422 branch is untested (stub only returns `NotFound`; needs a Postgres-backed edit-to-unknown-material). (b) Dead `RepositoryError::Duplicate(String)` variant in the spools SPI (copied from materials; spools have no unique name — unreachable, nonsensical message). (c) Detail 404 reuses the `spools.edit.not_found` i18n key instead of a `spools.detail.not_found`. (d) A couple of French localization tests use weak `contains(A) || contains(B)` assertions. | All cosmetic / test-quality; behaviour is correct. | Small readability/coverage gaps accumulate. | low — batch in a future cleanup pass. |
| TD-008 | **`Density::new` reuses `NonPositiveDensity` for non-finite input** (`crates/domain/src/materials/model.rs`). A `NaN`/`inf` density is correctly rejected but the message reads "density must be > 0 g/cm³, got NaN" — misleading. | Behaviourally safe (input rejected); message only. | Confusing error text if a non-finite density is ever surfaced to a user/log. | low — add a distinct `DomainError` variant / message for the non-finite case. |
| TD-009 | **Create/edit form-validation errors (422 on blank name, 409 on duplicate name) are not surfaced in the UI** for both **materials** (`crates/app/src/web/materials.rs`) and **locations** (`crates/app/src/web/locations.rs`) create/edit handlers. They return unwrapped text bodies (`String`/`format!`), and htmx's default `responseHandling` never swaps non-2xx bodies into the DOM — so the operator sees no feedback beyond the request silently failing. Only the locations blocked-delete 409 has inline feedback, via the htmx `response-targets` extension routing that one control's 409 to a dedicated `#locations-msg` slot (`hx-target-409` in `_location_row.html`); create/edit were deliberately left out of that fix to keep its scope tight. | An earlier fix attempt swapped ALL 4xx bodies globally via a `base.html` `<meta name="htmx-config">` override, but the 4xx handlers return plain text targeting `hx-target="closest tr"`/`#…-table-body`, not `<tr>` fragments — swapping bare text into a table corrupts the DOM. Reverted in favour of the scoped `response-targets` opt-in. | Blank-field/duplicate-name mistakes on create/edit fail with no visible feedback (single-user, self-hosted; low real-world friction). | low — wire `hx-target-422`/`hx-target-409` (or a shared `hx-target-error`) per form the same way the locations delete control does, once this pattern is needed for more than one control. |
| TD-010 | **`list_with_spool_counts` O(n) full-table rescan used for a single-row read**: `spool_count_for()` (`crates/app/src/web/locations.rs`, called after a successful edit to recover the row's count) re-lists every location and re-counts every location's spools just to find one row. | Acceptable at this app's scale (self-hosted, small location counts); no cheap single-location count read-model exists yet. The blocked-delete 409 path no longer does this — it now reads `count` directly off `DomainError::LocationInUse`, so only the post-edit path remains. | An O(n²)-ish (n locations × per-location `count_spools` query) cost on every edit as location counts grow; negligible at expected scale. | low — add a single-location count method to `LocationRepository`/`LocationsUseCases` if this ever needs to scale. |

| TD-004 | **No `MaterialName` invariant + `Density` accepts non-finite** (materials slice). `NewMaterial.name: String` has no non-blank/length invariant (a whitespace/empty name can be inserted; only `UNIQUE` caps it at one). `Density::new` checks only `> 0.0`, so `density=inf` from the form passes and persists. | Display-only in v1 (no functional harm); the add-form is operator-only, single-user. | Data-quality gaps; `Density::INFINITY` breaks the future `spools` Remaining-Length divide-by-density. | **Resolved 2026-07-06** — `Density::new` now requires `value.is_finite() && value > 0.0` (still returns `DomainError::NonPositiveDensity`). Added a validated `MaterialName` newtype (`crates/domain/src/materials/model.rs`) that trims and rejects empty/whitespace-only input, returning a new `DomainError::BlankMaterialName` variant. Wired `MaterialName` into `NewMaterial`/`Material`, rippled through `seed.rs`, the SQLx adapter (`crates/app/src/persistence/materials.rs`), the in-memory stub, use cases, and the htmx `MaterialForm`/`MaterialView` in `crates/app/src/web/materials.rs`. |
