| TD-001 | **Home-grown i18n** (`web/i18n.rs`): ~40-line JSON-catalog lookup with locale‚Üídefault‚Üíkey fallback. No pluralization, gender, or `{var}` interpolation. | Does exactly what ADR-0001 needs today (key‚Üístring); zero deps; YAGNI. Reviewed crates: **fluent-templates** (Mozilla Fluent/ICU ‚Äî plurals, gender, interpolation, `.ftl` files; heaviest, most capable) and **rust-i18n** (compile-time macro, JSON/YAML/TOML catalogs, `t!()` macro; lighter but macro-bound). | When a real string needs a plural or an interpolated value, the custom layer must grow or be swapped ‚Äî a swap touches every template's `t()` call. | low ‚Äî revisit when the first pluralized/interpolated string appears (likely a slice with counts, e.g. "N bobines"). |
| TD-002 | **Basic observability**: startup uses `tracing` with a simple subscriber; no structured request spans, no metrics, no log levels wired to config. | Foundation needs *a* real logger (not `println!`); full observability is not needed pre-v1. | Debugging a deployed instance is coarse until request-scoped tracing + levels land. | med ‚Äî add `tower-http` trace layer spans + config-driven level when the app is actually deployed to k3s. |
| TD-003 | **`update()` of an unknown material id is a silent 200 no-op** (materials slice). The SQLx `MaterialRepository::update` doesn't check `rows_affected`, so `PUT /materials/{unknown-id}` runs a 0-row UPDATE and returns a 200 row fragment as if it persisted. The in-memory stub instead returns `Backend("no such id")` (‚Üí500), so stub-backed tests don't catch the divergence. | Not reachable from the happy-path htmx UI (ids always come from a rendered row); no data loss; final review cleared it as deferrable. | A future non-UI caller (API, bulk edit) gets a false success; stub/prod behaviour disagree. | med ‚Äî add a `RepositoryError::NotFound` variant, check `rows_affected == 0` in the SQLx `update`, map to **404** in the web `edit` handler, and align the stub. |
| TD-004 | **No `MaterialName` invariant + `Density` accepts non-finite** (materials slice). `NewMaterial.name: String` has no non-blank/length invariant (a whitespace/empty name can be inserted; only `UNIQUE` caps it at one). `Density::new` checks only `> 0.0`, so `density=inf` from the form passes and persists. | Display-only in v1 (no functional harm); the add-form is operator-only, single-user. | Data-quality gaps; `Density::INFINITY` breaks the future `spools` Remaining-Length divide-by-density. | med ‚Äî introduce a validated `MaterialName` newtype; tighten `Density::new` to `v.is_finite() && v > 0.0` **before** the `spools` slice consumes density. |
| TD-005 | **Raw internal error strings returned to the client** (materials web handlers). 500 arms return `sqlx`/`tera` error text in the response body (info disclosure). | v1 is single-user, self-hosted, no auth ‚Äî the only client is the trusted operator; practical exposure is nil. | Leaks backend/SQL/template internals once the surface is ever exposed beyond the operator. | low ‚Äî map 500s to a generic message, log the detail server-side; do this if/when the app is exposed beyond localhost. |
