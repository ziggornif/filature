# Accepted Risks

> Risks that have been reviewed and deliberately accepted rather than mitigated.
> Each entry is a traceable decision — not a forgotten issue.
> Risks that meet the ADR criteria (hard to reverse, surprising, real trade-off) go in docs/adr/ instead.

| ID | Risk | Rationale for acceptance | Review date | Owner |
|-----|------|--------------------------|-------------|-------|
| AR-001 | **No per-user authorization; single shared credential.** | ~~Every route is fully open.~~ **Partially mitigated 2026-07-12 (slice 08)** — a demo-auth gate (`web::auth`) now sits in front of the whole app: a default-deny login middleware (allowlist = `/login` only, `/static` included) + a single operator credential (`[auth]` config, argon2id `password_hash`) + a per-boot session cookie. The gate is **always active in production** (missing `[auth]` fails the boot). It is **authentication only** — one credential, no roles/authz/IDOR checks, because the deployment is single-operator with no multi-tenancy or PII. Still deploy behind a reverse proxy enforcing TLS. Promote to an ADR + real authz if the app ever becomes multi-user. | 2026-07-12 | ziggornif |
| AR-002 | **Internal error detail leaks to the client.** 500-path handlers return `e.to_string()` (incl. raw sqlx text) in the response body. | ~~Low impact under AR-001 (the only client is the operator). Accepted as an interim state.~~ **Resolved 2026-07-10** — every web 500 path now routes through `web::router::internal_error`, which logs the detail with `tracing::error!` server-side and returns a generic `"Internal server error"` body. No raw error text reaches the client. Also closes TD-005. | 2026-07-10 | ziggornif |
| AR-003 | **Unbounded list queries** (spools/materials have no pagination). | Bounded in practice — a personal/micro-business filament inventory is hundreds of rows, not millions. Latent scaling concern, not a live DoS. Add pagination if inventory size ever warrants it. | 2026-07-10 | ziggornif |
| AR-004 | **No audit log** of stock mutations. | Single operator; no second party to dispute an action, so repudiation has no meaning here. Reconsider if the tool ever gains multiple users. | 2026-07-10 | ziggornif |
| AR-005 | **No brute-force / rate limiting on `/login`** (slice 08). | Single demo credential. Two distinct vectors: (1) **online password guessing** — argon2id's per-attempt cost (~tens of ms) meaningfully caps the guess rate; (2) **resource-exhaustion DoS** — argon2id's memory-hardness (~19 MiB/verify) *amplifies* a concurrent-login flood rather than mitigating it, so "argon2 is slow" is NOT a control here. **Mitigated (not just accepted):** a `LOGIN_VERIFY_CONCURRENCY` semaphore in `web::auth` bounds in-flight verifies (memory capped at ~19 MiB × N, excess attempts queue as cheap futures). Instances also sit behind a reverse proxy where a WAF/rate-limit can be added. Add app-level lockout only if a public demo shows abuse. | 2026-07-12 | ziggornif |
| AR-006 | **Session cookie not marked `Secure`; single shared session token** (slice 08). | The app terminates no TLS (an external proxy does) and can't reliably detect the outer scheme, so `Secure` is omitted to avoid breaking plain-HTTP local runs; the cookie is `HttpOnly` + `SameSite=Lax`, and TLS is expected end-to-end via the proxy. The token is one shared per-boot value (not per-session) — acceptable for a single demo user; a restart invalidates it. Revisit (per-session tokens, `Secure` via trusted `X-Forwarded-Proto`) if the app becomes multi-user or public. | 2026-07-12 | ziggornif |

| AR-007 | **No CSRF token on state-changing POSTs, incl. the destructive instance import** (`POST /settings/import` wipes + recreates all data). | Mitigated, not just accepted: the session cookie is `SameSite=Lax` (AR-006), which blocks cross-site cookie-bearing POSTs; the import additionally requires an explicit `confirm_replace=yes` field **and** a valid `multipart/form-data` file upload (not forgeable by a simple cross-site auto-submit), is bounded to 1 MiB, and validates a closed `filature-instance` v1 schema before touching the DB. Single-operator, self-hosted behind a TLS-terminating reverse proxy. Add per-form CSRF tokens if the app ever becomes multi-user or public. | 2026-07-13 | ziggornif |

| AR-009 | **Two RustSec advisories ignored in `deny.toml`**, both transitive via `rumqttc 0.25` (latest release; Bambu MQTT, slice 22b), unfixable without an upstream bump. **RUSTSEC-2026-0049** (`rustls-webpki 0.102` — CRLs not treated as authoritative by distribution point): the Bambu adapter deliberately bypasses all certificate verification (AR-008), so no CRL-matching path is ever exercised on this dependency; the app's other TLS (reqwest, 22a) resolves `rustls-webpki 0.103.13`, which is unaffected. **RUSTSEC-2025-0134** (`rustls-pemfile` unmaintained): pulled transitively, the app loads no PEM itself — an unmaintained flag, not a vulnerability. | Re-evaluate on every `rumqttc` release; drop the ignores as soon as a version ships with updated rustls deps. | 2026-07-22 | ziggornif |
| AR-008 | **Bambu LAN MQTT: TLS server identity not verified** (slice 22b). The printer presents a self-signed **X.509 v1** certificate that webpki cannot even parse for handshake-signature verification (`UnsupportedCertVersion`, validated against the real machine 2026-07-22), so the Bambu adapter skips both chain and handshake-signature verification. | A LAN MITM could impersonate the printer on this link (read-only status probe; the access code would transit to the impostor). Accepted because: (1) every Bambu LAN integration does the same (`mosquitto --insecure`, Home Assistant); (2) the link is confined to the trusted home LAN and the app sits behind the reverse proxy; (3) the bypass is scoped strictly to the Bambu adapter's TLS config — PrusaLink/Moonraker HTTP clients keep full rustls verification (slice 22a). Revisit if Bambu ever ships verifiable certs (pin-on-first-use would be the upgrade path). | 2026-07-22 | ziggornif |

> AR-001 was the load-bearing decision. Auth has since been built *into* the app
> as the demo-auth gate — see **ADR-0005** for the design (separate `AuthConfig`,
> outer-layer always-active enforcement, argon2id credential, per-boot shared
> session token). AR-001 now covers the residual "authentication only, no
> per-user authz" posture; AR-005/AR-006 cover the gate's known limitations.
