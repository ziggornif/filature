# Threat Model

> Generated by `security-review` guide mode, filled from the code as of 2026-07-10.
> Update when entry points, data flows, or trust boundaries change.

**System:** Filature — self-hosted 3D-printing filament stock manager. Single Rust
binary (Axum/SQLx/Tera/htmx) + PostgreSQL. Single operator, personal + "Zig Factory"
micro-business use. No multi-tenancy, no public sign-up. Humidity/MQTT is deferred
(no sensors, no MQTT surface in v0–v1).

## Assets worth protecting

- **Stock data** (spools, materials, locations, prices paid) — integrity and
  availability matter; confidentiality is low-stakes (no PII, no credentials, no
  payment data). The `price_paid` field is the only mildly sensitive value.
- **The database connection string** (`database.url`) — the one real secret.
  Loaded at the adapter edge from `filature.toml` / `FILATURE_*` env, never in the
  domain, never committed (`.env`, `*.local.toml` gitignored; no `filature.toml`
  in the repo).
- **Availability of the service** to its single operator.

## Trust boundaries

The hexagon is the trust-boundary diagram:

- **Driving (API) adapter — `crates/app/src/web/`** — the only inbound attack
  surface. Everything arriving here (HTTP path, query, form body, cookies) is
  untrusted until validated.
- **Domain — `crates/domain/`** — assumes its inputs are already legitimate.
  Pure, no I/O, no framework. Validation happens *before* a use case is called.
- **Driven (SPI) adapter — `crates/app/src/persistence/`** — outbound to
  PostgreSQL. Injection risk lives here; mitigated by parameterization.
- **Secrets/infra** — config load (`config.rs`), DB pool. Outside the domain.

There is **no authentication boundary**: any client that can reach the bound
socket has full read/write access to every route. See AR-001 in
[accepted-risks.md](accepted-risks.md).

## Entry points

All routes are unauthenticated HTML/htmx endpoints. Input validation is done in
the driving adapter (form → domain newtype), which rejects bad input with `422`
and a localized message rather than panicking or 500-ing.

| Route | Method | Untrusted input | Validation |
|---|---|---|---|
| `/` | GET | `lang`/`theme` cookies | unknown locale → default; theme enum-parsed |
| `/static/{*path}` | GET | path | served from embedded `RustEmbed` bundle only — no filesystem access, path can't escape the bundle |
| `/materials` | GET/POST | form (name, density, temps, sensitivity) | `MaterialName`/`Density`/`Sensitivity` newtypes; 422 on bad |
| `/materials/{id}` | PUT | path id + form | as above; unknown id → repo error |
| `/spools`, `/spools/rows` | GET | query (material_id, status, sort) | status/sort parsed to enums (unknown ignored); material_id parameterized |
| `/spools` | POST | form (colour hex/name, diameter, weight, price, location) | `Colour`/`Diameter`/`Grams`/`Money`/`LocationId`; 422 on bad |
| `/spools/{id}` | GET/PUT | path id + form | as above; FK violation → `UnknownMaterial`/`UnknownLocation` |
| `/spools/{id}/{remaining,consume,archive,restore,location}` | POST | path id + form | `Grams` parse; lifecycle rules in domain; 422/404 mapped |

## STRIDE analysis

- **Spoofing** — no identity to spoof; there is no auth. Any caller is
  effectively "the operator". This is the design (single-user, trusted network)
  and the top accepted risk (AR-001).
- **Tampering** — in flight: no TLS is terminated by the app (expected behind a
  reverse proxy or on localhost — AR-001). At rest: writes go only through
  parameterized SQLx statements; the domain enforces invariants (weight ≤ net,
  archived-spool lifecycle) before persistence. No mass-assignment: forms map
  field-by-field into typed newtypes, never blanket-deserialized into a domain
  entity.
- **Repudiation** — no audit log. Acceptable for a single-operator tool; there is
  no second party to dispute an action. Noted, not mitigated (AR-004).
- **Information disclosure** — **finding:** several handlers return
  `e.to_string()` (including `RepositoryError::Backend`, which wraps the raw sqlx
  error text) directly to the client on the 500 path. Internal detail (SQL,
  schema, driver messages) can leak to whoever holds the socket. Low impact given
  AR-001 (the only client is the operator) but it should be fixed — log
  server-side, return a generic message. See "Top risks" #2.
- **Denial of service** — input is bounded: Axum's default 2 MB body limit
  applies; form fields parse into small typed values. The spool/material lists are
  **unbounded** (no pagination) — bounded in practice for a personal inventory
  (hundreds of rows), a latent scaling risk not a live one (AR-003). DB pool uses
  SQLx defaults (10 connections, 30 s acquire timeout) — adequate for one user.
  No recursion, no unbounded outbound I/O (the app makes no outbound HTTP → no
  SSRF surface).
- **Elevation of privilege** — no privilege levels exist (no roles, no auth), so
  there is nothing to elevate *to*. IDOR is not applicable: there is only one
  principal, and every object legitimately belongs to it.

## Top risks and mitigations

Worst-first.

1. **Unauthenticated full access (AR-001).** *Likelihood:* certain if the socket
   is reachable by an untrusted party. *Impact:* full read/write of all stock
   data. *Mitigation:* bind to `127.0.0.1` or a trusted LAN interface; put behind
   a reverse proxy enforcing TLS + auth (basic-auth/OIDC) before any exposure
   beyond localhost. *Residual:* accepted for the trusted-network deployment model
   — recorded as AR-001, revisit if the app is ever exposed to the internet.
2. **Internal-error disclosure to client.** *Likelihood:* on any backend/DB
   error. *Impact:* leaks SQL/driver internals; low given AR-001. *Mitigation
   (recommended fix):* map the 500 paths to a generic localized message and
   `tracing::error!` the detail server-side, instead of `e.to_string()` in the
   response body. Not yet implemented — candidate for a small follow-up slice.
3. **No secret scanning in the pipeline.** *Likelihood:* human error (pasting the
   DB URL into a committed file). *Impact:* credential leak. *Mitigation:* add
   `gitleaks` to CI (and ideally pre-commit) — the one computational security
   sensor the harness calls for that was missing. *Status:* added to CI
   (`.github/workflows/ci.yml`, `secrets` job).

## Relied-upon controls (don't regress these)

- **Parameterized SQL only** — every query uses `sqlx::query!`; the dynamic sort
  is enum-dispatched to static query strings, never string-built. No
  `format!`-into-SQL anywhere.
- **Tera autoescaping** — templates are named `*.html`, so Tera HTML-escapes all
  interpolated values by default. User-controlled strings (colour name, material
  name, price) are rendered through this; do not introduce `| safe` on
  user-supplied data.
- **`cargo-deny`** — advisories + licenses + banned-source policy, run in CI on
  every push/PR and weekly on a schedule (catches new CVEs against unchanged code).
- **Domain purity + slice isolation** — enforced by `tools/check-*.sh` in CI;
  keeps secrets and I/O out of the domain by construction.
